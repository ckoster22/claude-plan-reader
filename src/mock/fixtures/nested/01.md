# Sub-Plan 01: Engine Scaffold, Body Model & Render Loop

> Part of the **Chompy Asteroids** master plan (web 2D asteroid game). This sub-plan builds the
> greenfield project scaffold and the rendering spine ONLY. Physics, collisions, gravity, input,
> and world population are later sub-plans and are explicit non-goals here.

## Context

The repo is greenfield: the working directory holds only `.plan-tree/` metadata and a
single-file Canvas prototype at `.plan-tree/prototype/index.html` (~570 lines, all globals in one
IIFE). That prototype is the **executable spec**. There is no `package.json`, source tree, build
tooling, engine, test runner, or git.

This sub-plan stands up the project and ports the prototype's **render spine** ŌĆö the parts that
draw the world ŌĆö onto a PixiJS render-only foundation, while establishing the
`Workflows / Operations / Connectors` module seam the prototype lacks and **freezing the canonical
`Body` contract** that every later sub-plan depends on. Rendering is driven by mock/stub bodies;
no simulation runs yet.

## Engine decision (confirmed): PixiJS, render-only

The prototype's physics is hand-rolled and engine-agnostic. We keep that custom sim (built in
later sub-plans) and use **PixiJS purely as the GPU render adapter**. Rationale recorded in the
repo README:
- Phaser would bundle Arcade/Matter physics that directly fight the bespoke velocity-regime
  collision model ŌĆö unused weight and conceptual conflict.
- PixiJS gives batched GPU rendering with no physics opinion; asteroids map cleanly to a
  `PIXI.Graphics` cleared+redrawn per frame (silhouettes mutate at runtime via dents, and the
  soft-body squash is a per-body affine ŌĆö both native to Pixi's transform/Graphics API).

## Tech stack

- **Vite** (vanilla-js template) as bundler + dev server (the visual verification harness).
- **PixiJS v8** (`pixi.js`) as the render adapter. **v8 API specifics that MUST be used (v7 forms
  are removed and will break):**
  - Bootstrap is async: `const app = new Application(); await app.init({ resizeTo: window,
    background: 0x05060a, antialias: true });` then mount **`app.canvas`** (NOT `app.view`).
  - Graphics verbs: `g.clear(); g.poly(points); g.fill({ color, alpha }); g.stroke({ width,
    color, alpha })` ŌĆö the v7 `beginFill/lineStyle/drawPolygon/endFill` are gone.
  - Blend mode is a **string**: `g.blendMode = 'add'` (maps the prototype's
    `globalCompositeOperation='lighter'`); there is no `BLEND_MODES.ADD` enum in v8.
  - Affine transforms set directly via `Matrix` + `container.setFromMatrix(matrix)` (see squash).
  - Color: do **not** rely on Pixi parsing `hsl()` CSS strings. Convert HSLŌåÆRGB manually (small
    helper) so the prototype's exact fill-lightness formula `l/100*40+18` (line 483) is preserved
    verbatim ŌĆö the color *formula*, not just the hue, is load-bearing.
- **Vitest** installed and configured now (scaffold only) so Sub-Plan 02 has a test runner ready;
  this sub-plan adds only smoke + the affine numerical test (see Verification), not physics tests.
- **git** initialized with a Node `.gitignore` (ignore `node_modules`, `dist`).

## The frozen `Body` contract (`src/domain/Body.js`)

Port `makeAst` (prototype lines 120ŌĆō129) verbatim in shape. `makeBody({...})` returns:

| field | kind | meaning / source |
|-------|------|------------------|
| `id` | sim | unique id (monotonic counter) |
| `player` | sim flag, read by render | true for the player body (render tints + center dot) |
| `dead` | sim | death/removal flag (unused by render this plan) |
| `x`, `y` | render + sim | world-space position |
| `vx`, `vy` | sim | velocity (unused by render this plan; present in contract) |
| `m` | sim + render input | mass; render derives radius from it |
| `r` | **derived, NOT stored** | always `MASS_TO_R(m)` ŌĆö a function, never a field |
| `rot`, `vr` | render (sim updates vr) | rotation angle and angular velocity |
| `verts[]` | render + sim | 9 radial multipliers (~0.78ŌĆō1.18); silhouette, mutated by dents later |
| `hue` | render | base color hue (player=195, drifters 20ŌĆō50) |
| `def`, `dvel`, `dax`, `day` | render + sim | soft-body squash magnitude, spring velocity, world axis (x,y) |
| `flash` | render + sim | contact/crack rim-flash intensity, fades each frame |

Notes baked into the file as comments:
- `r` is **derived** via `MASS_TO_R` (config) ŌĆö do not add a stored `r`; later plans must not
  introduce one.
- `player` is a sim-owned flag the renderer reads ŌĆö it straddles layers; documented as such.
- `makeBody` uses the finite-guard (`fin`) on numeric inputs (port the prototype's NaN guard) and
  defaults a sensible mass so a malformed stub can't poison rendering.
- **Factory must accept overrides for all fields, not just positional `(m,x,y,vx,vy,player)`.**
  The prototype constructs with `makeAst(...)` then **mutates** `def/dax/day/vr/verts/rot/hue`
  afterward (e.g. fracture fragments set `def/dax/day` at creation, line ~408). To keep the
  contract genuinely *frozen*, `makeBody` takes an options object so Sub-Plans 02/03 can set those
  at construction without amending the factory. Rest defaults are explicit: `def:0, dvel:0,
  dax:1, day:0, flash:0`. The contract documents that post-construction mutation of these fields
  is the established pattern (so later plans need not widen the type).

## Rendering port details (Canvas-2D ŌåÆ PixiJS)

`Connectors/PixiRenderer.js` reproduces the prototype's `render()` + `drawAst()` (lines 461ŌĆō540):
- A **world container** whose `scale = cam.z` and `position = (-cam.x*cam.z, -cam.y*cam.z)`
  reproduces `scale(cam.z); translate(-cam.x,-cam.y)` (line 527): Pixi computes
  `screen = world*cam.z - cam*cam.z = (world - cam)*cam.z`, matching the prototype. Note the
  prototype camera is **top-left anchored, not centered** (`viewRect` treats `cam.x` as the left
  edge; `seed` initializes `cam.x = -W/2/cam.z, cam.y = -H/2/cam.z`, line ~179). `mockWorld`/the
  bootstrap MUST seed this same offset or the designated player will not be centered (verification
  step 2 fails otherwise). The worldŌåÆscreen mapping is verified by a **numerical** hand-checked
  coordinate (see Verification), not by eye.
- Each body owns a persistent `PIXI.Graphics`; per frame: `clear()`, rebuild the `verts.length`-gon
  (lines 478ŌĆō481; the angle divisor is `verts.length`, not a hardcoded 9), `fill({color, alpha})`,
  `stroke({width, color, alpha})` (lines 482ŌĆō485, v8 verbs). Color via the manual HSLŌåÆRGB helper
  preserving the exact `hsl(${hue} ${sat}% ${l/100*40+18}%)` formula. The big-body purple
  gravity-halo gradient (lines 475ŌĆō477) is **deferred to a later plan** (keys on `m>500`, a physics
  concern) ŌĆö render spine draws the body only.
- **Soft-body squash ŌĆö set the local matrix DIRECTLY (do not decompose into rotation/scale).** The
  canvas stacks `rotate(ang)┬Ęscale(1-def,1+def*0.5)┬Ęrotate(-ang)┬Ęrotate(rot)` on one matrix; the
  squash block is an anisotropic scale about a world-space axis, i.e. `M_squash = R(ang)┬ĘS┬ĘR(-ang)`
  ŌĆö a sandwiched rotate┬Ęscale┬ĘrotateŌü╗┬╣ that a single Pixi node's `rotation`/`scale` properties
  **cannot** express (and a naive two-node nest reproduces a *rotated* silhouette, not a squashed
  one ŌĆö wrong). Instead: a **deform container** gets its local transform assigned via
  `setFromMatrix(squashMatrix(def, dax, day))` where `squashMatrix` builds the exact 2├Ś3
  `R(ang)┬ĘS┬ĘR(-ang)` (`ang = atan2(day,dax)`), and its **child** body `Graphics` carries `rot` as
  its own `rotation`. This is element-for-element identical to the canvas math. `squashMatrix`
  lives in `src/math/affine.js` and is unit-tested numerically. When `def Ōēż 0.001`, skip the
  deform matrix (identity), matching the prototype's guard (line 468).
- `flash` rim glow (lines 487ŌĆō491) ŌåÆ an additive (`blendMode='add'`) restroke at `alpha=flash`.
- Player center dot (line 493) and player tint (line 482ŌĆō485) preserved.
- **Draw order via z-index, not re-parenting:** big bodies render on top (prototype sorts ascending
  by mass each frame, line 530). To keep Graphics *persistent* (required by the flat-memory check),
  set `worldContainer.sortableChildren = true` and each body node's `zIndex = body.m` per frame ŌĆö
  do NOT remove/re-add children every frame.
- Particles and shock rings (lines 528, 532ŌĆō538) are collision FX ŌåÆ **stubbed empty arrays** this
  plan; the renderer iterates them (zero items) so the integration point exists.

`Connectors/Starfield.js` ports `starLayers`/`starPos`/`drawStars` (lines 498ŌĆō522) into a
**screen-space** container that is NOT under the world transform: 3 layers, per-layer modulo
tiling driven by `cam.x*cam.z*depth` (lines 511ŌĆō518). Faithful port = **one `Graphics` cleared and
redrawn each frame** with `rect(...).fill(...)` per star (mirrors the prototype's per-frame
screen-space redraw); avoid `ParticleContainer` (v8 holds `Particle`/sprite objects, not arbitrary
rects ŌĆö an API mismatch). The tile span is the current screen `W,H`, so `starPos` positions
(`frac*W`) must be **recomputed on resize** (the pattern re-lays out ŌĆö acceptable, but must follow
live `W,H`). Must tile seamlessly as the camera moves arbitrarily far in any direction
(positive-modulo math `(((v)%t)+t)%t` ported exactly).

`Connectors/Hud.js` ports the DOM HUD (markup lines 26ŌĆō35, updates 542ŌĆō559) but reads from a plain
**view-model object** the loop hands it (`{total, biggest, eaten, merged, shatter, mass, rank,
growthPct}`), not from globals. This plan feeds it mock/placeholder values (counts of mock bodies,
zeros for eaten/merged/shatter). The game-over overlay branch (lines 555ŌĆō559) is ported as dormant
markup (a later plan wires the death path).

## The loop shell (`src/Workflows/GameLoop.js` + `src/main.js`)

- Port the try/catch RAF loop (lines 562ŌĆō567) **verbatim in spirit**: each frame is wrapped so one
  thrown error is logged and the loop survives (`requestAnimationFrame` re-armed unconditionally).
- **Plumb `dt` now** even though no physics consumes it: compute `dt` from successive
  `performance.now()` timestamps, clamp it to a sane max (e.g. Ōēż ~2 frames to avoid spiral-of-death
  after a tab stall), and pass it into the injected `step(dt)` (no-op stub here) and a
  `tick(dt)` for the renderer's frame-time-independent bits. Master plan mandates a dt-based sim;
  establishing the dt source in the loop shell is in-scope so Sub-Plans 02/03 inherit it.
- **dt unit convention (decided now so 02/03 inherit it cleanly):** the prototype is
  frame-rate-dependent ŌĆö every integration and decay constant (`THRUST`, the deform spring,
  `flash*0.86`, cam lerp `0.06`) is **per-frame at ~60fps**, not per-second. To let those tuned
  constants port **verbatim and stay correct**, the loop passes a **normalized** delta
  `dtNorm = clamp(rawMs / 16.667, 0, 2)` (1.0 at 60fps; clamped to Ōēż2 frames to avoid a
  spiral-of-death after a tab stall). `step(dtNorm)`/`render(dtNorm)` receive this. Document the
  convention in `GameLoop.js`: physics multiplies per-frame deltas by `dtNorm`, so the prototype's
  numbers need no rescaling. (Raw ms is also available for any true wall-clock needs.)
- `main.js` creates the Pixi `Application` (**`await app.init(...)`**, v8 async), appends
  **`app.canvas`** to `#wrap`, builds the renderer/starfield/HUD, loads `mock/mockWorld.js`, wires a
  **window `resize`** handler (`app.renderer.resize` is automatic via `resizeTo: window`; the
  handler updates the cached `W,H` used by camera-centering and the starfield span), and starts the
  loop with `step = () => {}` (no-op) and `render = (dtNorm) => renderer.draw(mockBodies, cam, hudVM)`.

## Verification (visual + smoke; no physics tests this plan)

1. **Dev server runs:** `npm run dev` serves the app; the canvas shows the dark field, tiled
   starfield, and the mock asteroids with correct silhouettes/colors. No console errors.
2. **Camera follows + zooms:** with the seeded center offset the designated player stub is
   centered; the scripted drift shows the camera lerp-follow and the radius-based zoom path both
   fire. Removing the player stub: the `if(me)` guard prevents any thrown error AND the dormant
   game-over overlay markup appears (death-path placeholder, wired but inert) ŌĆö confirm both, since
   an unguarded no-player path would throw every frame and silently contradict check #1.
3. **Starfield tiles infinitely:** programmatically jump `cam.x/cam.y` to large +/- magnitudes
   (e.g. ┬▒1e6) and confirm the parallax layers still fill the screen with no gaps/seams and no
   NaN ŌĆö the positive-modulo tiling is correct in all directions.
4. **Affine correctness ŌĆö NUMERICAL, not visual (Vitest, no GPU):** the soft-body squash is a
   geometry transform, so per project rules visual "looks squashed" is NOT proof. Unit-test
   `squashMatrix(def, dax, day)` and the world transform: pick a body vertex, compute its final
   screen position (a) through the prototype's matrix math (`R(ang)┬ĘS┬ĘR(-ang)┬ĘR(rot)` then
   `(world-cam)*cam.z`) and (b) through the renderer's matrix pipeline, and **assert equality to
   tolerance**. Show the test fails if the squash is mis-built (e.g. dropping the inner `R(-ang)`,
   the exact two-node-nest bug). A visual screenshot of the squashed stub is a secondary sanity
   check only.
5. **Sustained run:** leave it running several minutes; the try/catch loop logs nothing, frame
   rate is stable, memory is flat (per-body Graphics reused via `clear()`+redraw and z-index
   reorder, never recreated or re-parented per frame).
6. **Smoke tests (Vitest):** `makeBody` returns all frozen fields with finite numbers, honors
   option overrides for `def/dax/day/verts/hue/rot/vr`, and applies the NaN guard (pass `NaN` mass
   ŌåÆ falls back); `MASS_TO_R` is monotonic increasing; `hslToRgb` reproduces the prototype's exact
   fill lightness `l/100*40+18`. These prove the harness works for Sub-Plan 02. Each assertion
   shown to fail if the guard/derivation/override is broken.

Chrome DevTools MCP may be used to screenshot the running canvas for the camera/starfield/squash
sanity checks; if unavailable, verify in a normal browser. The affine and contract correctness
claims rest on the Vitest assertions (#4, #6), not the screenshots.
