## Context

Recon found the build is **already a faithful port of the prototype's validated constants** — every
tunable in the prototype's `47–153` block matches `src/config.js` except the two deliberate
divergences (normalized exponential gravity `GRAV_EXP=1.6` + `M_REF=300`; momentum-exact merge,
which is `collision.js` code, not a constant). There is **no unintended drift to reconcile**.

**The one substantive defect (and the core of this sub-plan): gravity is effectively DEAD.** `G`
was ported verbatim (`0.012`) but the force law changed from the prototype's `G·m^1.0/(d²+S)` to
`G·(m/M_REF)^1.6/(d²+S)`. Normalizing by `M_REF=300` raised to `1.6` rescales force by
`(1/300)^1.6 ≈ 1/9300`, so the felt pull is ~4 orders of magnitude weaker than the prototype.
At `m=300, d²+S=22000` the current per-frame accel is ≈ `0.012·1/22000 ≈ 5.5e-7` — utterly
negligible. This **violates the master success criterion "exponential gravity visibly bends
drifter paths."** Fixing it is the deliverable.

## The G re-derivation (a calculation; USER DECISION: player-felt field)

General anchor: match src pull to prototype pull at anchor mass `m_a`:
```
G_new · (m_a/M_REF)^1.6 / (d²+S) = G_proto · m_a / (d²+S)
⇒  G_new = G_proto · M_REF^1.6 · m_a^(−0.6)      [M_REF=300, GRAV_EXP=1.6, G_proto=0.012]
```
**The user chose a PLAYER-FELT field (~G=8–12)** — gravity should be felt routinely while flying,
not only near giants. **Set `G = 10` in `src/config.js`** (the only constant change). This anchors
the prototype-level pull at `m_a ≈ 55` (`10 = 0.012·300^1.6·m_a^−0.6` ⇒ `m_a^0.6 ≈ 11.04` ⇒
`m_a ≈ 55`) — i.e. bodies from ~55 mass up feel prototype-level-or-stronger pull, so the player
(mass 100) and most mid-size rocks feel meaningful gravity, while giants still dominate
super-linearly (the requested "exponentially more gravity for large asteroids").

Properties (by calculation, documented in a config comment):
- **dtNorm caveat:** the per-frame magnitude is exact at `dtNorm=1` (60fps); `integrate` applies
  `v += a·dtNorm` (single-dt, no double-dt — verified), so the field matches the prototype's
  per-frame model at the design framerate. State this in the comment.
- **Differentiation preserved:** larger masses pull super-linearly more (e.g. `m=1400`:
  `(1400/300)^1.6 ≈ 11.76`); small drifters pull little — "ratio of pull grows with size."
- **Accel cap stays a backstop, not the active constraint** — verify in the test. Worst single-pair
  case `m=MASS_CEILING=4000` at touching range `d²+S≈22000`: accel ≈
  `10·(4000/300)^1.6/22000 ≈ 10·63.0/22000 ≈ 0.029` < `GRAV_ACC_CAP=0.05`. At `m=1400`: ≈ `0.0024`.
  So the cap engages ONLY for the very largest bodies at closest range (a desired slingshot-bound,
  not "every large interaction" — the master-plan worry is still avoided). `GRAV_ACC_CAP` and
  `GRAV_SOFTEN` need **no change**. NOTE the cap is on the *accumulated* per-body accel (sum over
  pairs), so coherent clustering of giants could saturate it — implausible at `MAX_BODIES=140` with
  the power-law distribution, but the long-run stability test (block 3) is the backstop that proves
  no runaway at the live density.
- **Uncapped-player runaway check:** the player has `velCap:Infinity` (only thrust is capped), so a
  strong field could in principle fling it. The per-frame gravity accel cap (0.05) bounds the
  build-up; block 3's test MUST assert the player's speed stays finite and bounded (well below a
  sane ceiling, e.g. a few × `VEL_CAP`) over thousands of frames near a giant — this is the real
  no-runaway proof under the stronger field.

(If the user finds G=10 too strong/weak after manual play, it is a one-line change; 8–12 is the
sanctioned range.)

## New test — `test/workflows/EndToEnd.test.js` (the falsifiable acceptance backstop)

Deterministic, headless (injected `mulberry32` from `test/operations/_rng.js` + `makeBody` +
`Body._resetIdCounter`), extending the `SimStepSmoke.test.js` patterns. Four blocks, each
falsifiable (break → red → restore, noted in a comment):

1. **Regime reachability (direct `resolveCollision`, gravity-independent):** hand-build comparable
   overlapping pairs with normal closing speed `vn` (i) `< V_MERGE`, (ii) strictly between, (iii)
   `≥ V_FRACTURE`; call `resolveCollision(a,b,budget,rng)` directly; assert `regime` is
   `'merge'`/`'chip'`/`'fracture'` respectively (and that fracture yields ≥2 real `fragments`).
   Proves all three regimes are reachable + distinct independent of the sim/gravity tuning.
   Falsify: assert the wrong regime for a band → red.
2. **Gravity visibly bends a drifter (the success criterion + cap-is-backstop) — PINNED geometry,
   not borderline:** seed ONE big body `m≈4000` fixed at origin and a single drifter (no thrust,
   no other bodies, `ISOLATED_VR` so no respawn) starting at a **perpendicular pass at `d≈150`**
   with a tangential velocity (~1–2/frame), stepped `makeSimStep` for **N≈80** frames. Assert the
   drifter's heading change is **≥ 3× the visibility threshold** (i.e. `≥ ~6°`, and at G=10 this
   setup deflects far more — roughly tens of degrees — so it's comfortably non-flaky; do NOT pick a
   far/fast/short pass that lands near 2°). ALSO assert the per-frame gravity-accel magnitude on the
   drifter stays `< GRAV_ACC_CAP` for the whole pass (single-pair → cap is a backstop). Falsify:
   set `G` tiny/`0` → no measurable bend → red (the test that catches the dead-gravity bug).
3. **Live-sim stability / no-runaway (respawn ON, real `liveVR`) — incl. the uncapped player:** full
   `makeSimStep` over MANY thousands of frames (e.g. 5000); assert in a final sweep (and spot-checks):
   all `body.x/y/vx/vy` finite (no NaN/Inf), `bodies.length ≤ MAX_BODIES`, `particles ≤ MAX_PARTICLES`,
   `shocks ≤ MAX_SHOCKS`, every **drifter** `hypot(vx,vy) ≤ VEL_CAP + ε`, **and the PLAYER (which has
   `velCap:Infinity`) stays finite and below a sane ceiling (e.g. `≤ 6×VEL_CAP`)** — the real
   no-runaway proof under the stronger G=10 field; population stays in a `POP` band. Run a second
   variant seeding the player adjacent to a giant to stress the uncapped-player path. Falsify: remove
   a cap / disable the VEL_CAP clamp → a bound is exceeded.
4. **Mass non-increasing — respawn OFF (`ISOLATED_VR`):** a hand-seeded isolated field stepped
   many frames; total field mass is **non-increasing within ε** (the size-mismatch path is
   mass-*bounded* not strictly conserving, so use non-increasing-within-tolerance, NOT strict
   equality — matches the 04.01/02 finding). **Do NOT** assert mass-non-increasing on the LIVE
   (respawn-ON) sim — `cullAndRespawn` legitimately injects fresh drifter mass; that trap is why
   blocks 3 and 4 use different viewRect modes. Falsify: inject mass in the seed mid-run → red.
