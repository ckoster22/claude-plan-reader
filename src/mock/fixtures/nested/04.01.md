## Step 2 — Triage the 3 reds (prove code-vs-test; INSTRUMENT before theorizing)

**MANDATORY FIRST STEP — do NOT trust any pre-written narrative (including this plan's):** for each
red seed, run the seed through `makeSimStep` once and **print the ACTUAL resolved pairs** — for
every overlapping pair, log `overlapping().hit`, the descriptor `regime`, `dead`, `survivors`
(`mNew`), and `fragments`, plus each body's **post-integrate** `(x,y,vx,vy)`. Reconcile what
actually fires against the test's stated invariant. The step-4 integrate pre-pass (`SimStep.js:60`
`gravityAccel` → `:65–80` `integrate`+spin+`softBodyStep`) mutates every body BEFORE the collision
loop (`SimStep.js:82+`): it clamps drifter speed to `VEL_CAP=4.0` AND gravity displaces positions
by whole pixels — so the colliding pair, the regime, and even WHICH bodies touch can differ
entirely from the seed's intent. (`applySurvivor` `SimStep.js:30–47` sets `body.m=s.mNew` verbatim
— the apply path itself does not create mass; total field mass is non-increasing in all 3 reds, so
none is a mass-creation bug — but that does NOT mean the seeds test what their names claim.)

- **RED 1 — `WorldManager.test.js:85` `>0.5` (genuine literal mis-derivation):**
  `P(below MASS_MIN+0.05·MASS_SPAN) = 0.05^(1/POWER_EXP) ≈ 0.488` for `POWER_EXP=4.2`; the `>0.5`
  literal is simply wrong. Code is correct. Fix the TEST to assert a **bound derived from
  `POWER_EXP`** (compute the expected fraction from the constant, assert within a sampling
  tolerance ≈`±0.03`), NOT a fresh literal — survives a 04.04 `POWER_EXP` retune. Keep the passing
  median assertion. Falsifiability check uses a LOCAL throwaway (a separate `powerMass`-with-exp=1
  in the test), NOT by editing the committed `config.js` constant.
- **RED 2 — `SimStep.test.js:120` `bodies.length<3` (seed goes separating):** verified by probe —
  post-integrate all three static bodies overlap but `relNormal ≈ +1.85e-7` (gravity gives the
  outer bodies a microscopic OUTWARD normal component), so `closing` is false and every pair takes
  SLOW+**separating** de-overlap → no merge, length stays 3. The no-inflation PRIMARY (`:117`)
  passes. Fix: **redesign the seed** to a concrete closing pair — give the bodies a real closing
  velocity well above the ~1e-7 gravity nudge AND keep `vn < V_MERGE=1.15` so it stays in the merge
  regime post-integrate; **assert the descriptor `regime==='merge'`** (not just the count drop)
  before asserting `bodies.length<3`. Keep both the no-inflation and merge-fired assertions.
  Falsifiable: remove the dead-guard → mass inflates → red.
- **RED 3 — `SimStep.test.js:84` `toBeCloseTo(388,6)` (the seed tests the WRONG collision —
  highest risk, the plan's earlier theory was FALSE):** probe-verified reality: post-integrate
  A is at `x≈-4.0`, B at `x≈57.5` → **A and B do NOT overlap (no A–B fracture ever happens)**;
  the only collision is **A–C** (coincident at origin), which is a **size-mismatch fracture**
  (ratio 200/60=3.33 ≥ `SAME_SIZE`): C shatters, A *grabs* `60·FRAG_MASS_LOSS=10.8` → A=210.8, B
  untouched=200, C→frags≈49.2, total ≈**460** (the observed `459.99…`). The test's expected
  `388 = (200+200)·0.82 + 60` assumes an A–B comparable fracture that physically never occurs.
  **Therefore: do NOT re-pin the number to 460** — that would make a test named/commented
  "`#6` deferred-buffer integrity / frags don't collide the frame they're born" actually assert a
  size-mismatch-grab outcome, an unfalsifiable name/behavior conflation (the exact anti-pattern
  this gate exists to prevent). Instead **REDESIGN the seed to genuinely exercise the deferred
  buffer**: A and B must still **overlap AFTER integrate** (seed A's closing speed ≤ `VEL_CAP=4`
  so it isn't clamped, and account for the per-frame x-displacement, or use a smaller `dtNorm`),
  C must NOT be coincident with A (move it clear), and the regime that fires must be the intended
  **comparable fracture**. Then assert the deferred-buffer invariant the test claims: fragments
  enter `bodies` only at commit (no fragment-vs-original collision the same frame) AND total field
  mass is non-increasing. Note: the per-event formula `summed frag mass = parentM·(1−FRAG_MASS_LOSS)`
  is the COMPARABLE-fracture invariant only — the size-mismatch path adds `grab` to the survivor —
  so use **total-field-mass-non-increasing** (seed-robust) as the conservation guard, not that
  formula, unless you've confirmed the realized regime is comparable fracture.
- **Real-bug escape hatch:** fix PRODUCTION code ONLY if, after instrumenting, a red's PRIMARY
  invariant is genuinely red OR a seed-independent trace shows the apply path drifts/creates mass.
  Recon + probes indicate neither holds — but the instrumentation (not assertion) is what decides.
- **Seed hygiene (all `test/workflows/*`):** a seeded drifter speed `> VEL_CAP=4.0` is a fiction
  (integrate clamps it); add an assertion/comment that seeded speeds are `≤ VEL_CAP` to be
  meaningful, so future seeds don't repeat the trap.

## Step 3 — Add the deterministic headless "sim-is-live" smoke

New `test/workflows/SimStepSmoke.test.js` (Vitest, deterministic via injected `mulberry32` +
`Body._resetIdCounter()`), driving `makeSimStep` over a real `WorldManager` (NO browser — recon
confirms `__chompy`/Pixi can't boot in Node and exposes no step/stop seam). Assert:
- (a) **sim advances:** after N steps a non-player body's `(x,y)` changed.
- (b) **controls + coasting (gravity-isolated):** the player's gravity is UNCAPPED, so nearby
  drifters would change its speed and defeat a naive "speed unchanged" check. Use an **isolated
  world — the player alone (no drifters)** so `gravityAccel` is zero: with `input.inputState.up=true`
  for K steps the player's `vy` decreases (moves −y); after releasing, the player's speed is
  **unchanged** over several steps (drag=1 / coasting). (Stub `{inputState}` — `applyThrust` reads
  it.) Alternatively assert "unchanged within a `GRAV_ACC_CAP`-bounded epsilon" if drifters are
  present — but isolated is cleaner and exactly tests coasting.
- (c) **collisions fire (isolated 2-body closing pair):** seed **exactly two** comparable bodies,
  overlapping and **closing** (2-body gravity is purely attractive → closing, so it won't flip
  separating like the RED-2 3-body case). Step once; assert a `world.counters` field
  (`merged`/`shatter`) incremented **and** the descriptor `regime` is the expected one. (Do NOT use
  a static or 3+-body hand-placed pile — that reproduces the RED-2/RED-3 gravity traps.)
- (d) **zero swallowed errors:** spy on `console.error` across the run, assert **0** calls — this
  catches a silently-throwing step that `GameLoop`'s try/catch (`GameLoop.js:46–52`,
  `console.error('frame error (recovered)…')`) would otherwise hide. (Drive `step(dtNorm)` directly
  AND, separately, exercise it through `startGameLoop` with an injected `now` + a spy to prove the
  loop path is clean.)
- (e) **bounds:** over a multi-hundred-step run, `bodies.length ≤ MAX_BODIES`, `particles ≤
  MAX_PARTICLES`, `shocks ≤ MAX_SHOCKS`, all positions finite, density ≈ `POP`.
- (f) **mass non-increasing:** total field mass across N steps is non-increasing within dust
  tolerance (never CREATES mass — the slow-drift guard bounds checks can't see).
Each assertion falsifiable (break the relevant code path → red → restore; note it in a comment).
