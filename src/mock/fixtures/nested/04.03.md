# Sub-Plan 04.03: Lose / Restart Flow

> Leaf child of Sub-Plan 04. Builds on the committed green baseline (HEAD `aa9cf82`, 96 tests).
> Make death legible (game-over copy driven from the HUD VM) and the game replayable in-place
> (restart without a page reload). Frozen layers (`src/Operations/*`, `src/domain/Body.js`,
> `src/math/affine.js`) are READ-ONLY. No tuning.

## Context

Lose-detection is already live: the player is collision-killed only (never culled —
`WorldManager.cullAndRespawn` exempts `player` at line 138, `if (A.player) return true;`), and
when the player leaves `world.bodies`, `buildHudVM`'s no-player branch returns `mass:null`, which
`Hud.update` already turns into a visible `#over` overlay. Two gaps remain:
1. **Game-over copy is hard-coded in the Connector.** `Hud.js` does not capture `#otitle`, and its
   `#osub` string is hard-coded reading only `vm.eaten` (`Hud.js:49–53`). The copy must come from
   the VM.
2. **Restart is a full page reload.** `index.html`'s `Play again` button is
   `onclick="location.reload()"`. Replace it with an in-place reset — but the reset state-inventory
   must be COMPLETE (an incomplete reset is worse than a reload).

**Parent adjustment (honored):** the game-over summary uses ONLY persisted `world.counters`
(eaten/merged/shatter) — NOT "final mass". Once the player is collision-killed it is absent from
`world.bodies`, so `buildHudVM` cannot read its mass, and there is no last-known-mass cache; do NOT
add mass-tracking to the committed SimStep/WorldManager baseline just to show it.

## Changes

### A. Game-over copy from the VM (3 small edits)
1. **`src/Workflows/hudVM.js`** — widen ONLY the no-player branch (line 37) to also return
   `title: 'Consumed.'` and `summary` built from the counters already in scope (`base.eaten`,
   `base.merged`, `base.shatter` — all `?? 0`-guarded). Example summary:
   `\`Eaten ${eaten} · Merged ${merged} · Shattered ${shatter}\``. The player branch is UNCHANGED
   (no `title`/`summary` there — their absence is how `Hud` knows not to render copy). Keep
   `mass:null/rank:'—'/growthPct:0` exactly (the `mass==null` overlay driver depends on it).
2. **`src/Connectors/Hud.js`** — (a) capture `#otitle` in the constructor element map (alongside
   `over`/`osub`, ~line 27); (b) in the overlay-shown branch (~lines 48–53), render FROM the VM,
   **`if`-guarded like every other Hud write** (`if (e.otitle) e.otitle.textContent = vm.title ??
   'Consumed.'; if (e.osub) e.osub.textContent = vm.summary ?? '';`), replacing the hard-coded
   `vm.eaten` string. (c) Leave the visibility driver
   (`hasPlayer = vm.mass !== null && vm.mass !== undefined`, lines 41/46/54) UNTOUCHED — keep the
   `mass==null` single source of truth, no one-shot flag.
3. **`index.html`** — `#otitle` already contains static "Consumed." text; leave the markup text
   (the VM now drives it, and the static text is a harmless fallback). The only index.html change
   is the button wiring (section B).

### B. In-place restart (replace `location.reload()`)
4. **`src/Workflows/WorldManager.js`** — add a `reset()` method (the single restart primitive; keep
   `spawn()` itself unchanged so the 96-test baseline doesn't regress):
   ```
   reset() { this.particles.length = 0; this.shocks.length = 0;
             this.counters.eaten = 0; this.counters.merged = 0; this.counters.shatter = 0; }
   ```
   (FX pools + counters are WorldManager-private state, so the reset belongs here, not inlined in
   main.js — keeps encapsulation and makes it headlessly testable. Name it `reset()`; it does NOT
   spawn — the caller sequences `reset()` then `spawn()`.)
5. **`src/Connectors/Input.js`** — add a public `clear()` that zeroes all four `inputState` flags
   (`up/down/left/right = false`); have the existing private `_onBlur` delegate to `clear()` (or
   duplicate the four assignments). This is the **highest-risk restart bug**: a button mouse-click
   does NOT blur the window, so a player who dies holding thrust and clicks Play-again would keep
   thrusting into the new game unless input is explicitly cleared.
6. **`src/main.js`** — add a `restart()` closure inside `main()` (captures `world`, `cam`, `input`,
   `screenW/screenH`), sequenced to match bootstrap order:
   `world.reset()` → `Body._resetIdCounter()` → `world.spawn()` → **`cam.z = 1`** → re-center cam
   (`cam.x = -screenW()/2/cam.z; cam.y = -screenH()/2/cam.z;`) → `input.clear()`.
   **`cam.z = 1` is REQUIRED and easy to miss:** the bootstrap re-center math is only correct when
   `cam.z===1` (as it is at boot). But by death `cam.z` is frozen at the last live zoom — a player
   who grew large leaves it pinned at the `0.7` floor (`updateCamera` is `if(me)`-guarded, so it
   stops updating once the player is culled). Re-centering with the stale `0.7` mis-centers the new
   small player → a visible ~1s zoom/center pop on every restart (the exact flow this sub-plan
   polishes). Reset `cam.z = 1` BEFORE the re-center division. Do **NOT** imperatively hide `#over`
   — the next `render`→`buildHudVM` sees the
   new player (`mass!=null`) and `Hud` hides it (avoids the two-way desync). Wire `restart()` to:
   - the `Play again` button: drop the inline `onclick`, give the button `id="again"` in
     `index.html`, and `document.getElementById('again').addEventListener('click', restart)` in main.js;
   - an `R` key: a SEPARATE `window.keydown` listener in main.js (`if (e.key === 'r' || e.key === 'R') restart()`)
     — do NOT add an `R` field to `Input.inputState` (restart is an edge event, not a per-frame
     state SimStep would have to ignore). Optionally add `restart` to `window.__chompy`.
   Canvas is already sized post-boot, so the degenerate-canvas guard isn't needed for a user-
   triggered restart; spawn-before-cam-recenter order is preserved regardless.

### Invariant to preserve (confirm, no change)
`cullAndRespawn` still exempts the player (`WorldManager.js:138`); the player dies by collision
only, never culling; the opening stays survivable. Mid-frame death is already safe (SimStep
re-resolves the player post-commit, caches no stale ref) — no work needed.

## Verification — falsifiable unit tests (headless, no main.js/Pixi import)

Construct `WorldManager`/`Input`/worlds directly (the established `test/workflows/*` pattern with
injected `mulberry32` + `makeBody`). The actual button/key wiring + cam visual are browser-only
(manual); assert restart's STATE EFFECTS by calling the reset primitives directly.

- **`WorldManager.reset()`** (new `test/workflows/WorldManager.test.js` cases): seed `particles`
  and `shocks` non-empty and `counters` non-zero → `reset()` → both pools `length===0` and all
  three counters `===0`. Falsify: leave `shocks` untouched / skip a counter → red.
- **restart reseed (sequence on a constructed world+input):** after a populated+culled world,
  `world.reset(); Body._resetIdCounter(); world.spawn();` → `bodies` has **exactly one**
  `player:true` at world `(0,0)`, population ~`POP` (≤ `MAX_BODIES`), counters zero, FX pools
  empty. Falsify: skip `reset()` → counters/FX persist → red.
- **deterministic player id/position after reset:** assert narrowly on the PLAYER body only (NOT
  "same whole-world shape" — a reused seeded rng advances its internal state across two spawns, so
  the drifters legitimately differ). After `_resetIdCounter(); spawn();` the player has the same id
  (e.g. `0`) and is at `(0,0)`. Falsify: drop `_resetIdCounter()` → the player id drifts → red.
  (This tests the id-reset invariant; do not over-claim it proves full spawn determinism.)
- **`Input.clear()`** (new `test/workflows/Input.test.js` — construct `Input` with a stub target
  `{addEventListener(){}, removeEventListener(){}}`): set all four `inputState` flags true →
  `clear()` → all four false. Falsify: clear only `up` → red. (Also assert `_onBlur` clears all
  four, since it now delegates.)
- **hudVM dead-branch summary** (extend `test/workflows/hudVM.test.js` no-player block, lines
  116–127): a no-player world with `counters {eaten:3,merged:2,shatter:1}` →
  `vm.title === 'Consumed.'` and `vm.summary` contains `3`, `2`, `1` (the counts). The existing
  `mass===null/rank==='—'/growthPct===0` assertions MUST still pass. Falsify: break the summary
  template (omit a count) → red. Also assert the PLAYER branch carries NO `title`/`summary` (so
  `Hud` only renders copy when dead).
- **overlay driver unchanged:** a hudVM unit assertion that player-present → `mass!=null` and
  no-player → `mass===null` (the `Hud` `#over` toggle contract). Falsify: return `mass:0` on death
  → red (already covered by 04.02's no-player test; keep it green).
