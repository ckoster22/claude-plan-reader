# Sub-Plan 04 (DECOMPOSITION): HUD, Integration & Tuning

> Final sub-plan of the **Chompy Asteroids** master plan. It is too large for one plan, so it is
> split into four sequential child sub-plans (04.01 → 04.04). Each consumes the prior's result.
> Scope: integration + surfacing + balancing ONLY — no new subsystems. Frozen lower layers
> (`src/Operations/*`, `src/domain/Body.js`, `src/math/affine.js`) are read-only.

## Context

Sub-Plans 01–03 produced the full vertical slice: render spine (PixiJS v8), pure physics
(`src/Operations/*`), and the integration + infinite world (`WorldManager`, `SimStep`, `Input`,
wired through `main.js` + `GameLoop`). **Important state note (from the Sub-Plan 03 agent's actual
report, after a session-limit cutoff):** the work is **complete on disk but UNCOMMITTED** — HEAD
is still the Sub-Plan 02 commit `b9f84e5`. `npm run build` succeeds. The test suite is
**77 passing / 3 failing (80 total)**: all 59 prior Operations/affine/body tests stay green; the
3 reds are all in the new `test/workflows/*` suites and the agent characterized them as
**test-calibration** issues, NOT implementation defects:
  1. `WorldManager` power-law — `belowThreshold/N ≈ 0.483` vs a `> 0.5` assertion (the `>0.5`
     boundary was mis-derived for `POWER_EXP=4.2`; small-weighting itself holds).
  2. `SimStep` dead-guard `#6b` — the real invariant (no mass inflation) PASSES; the secondary
     `bodies.length < 3` ("a merge fired") fails because the seeded slow geometry de-overlapped
     instead of merging (wrong seed, not wrong code).
  3. `SimStep` deferred-buffer `#6` — a fracture mass-conservation `toBeCloseTo` expectation
     mismatches `collision.js`'s exact dust accounting.
Securing, **rigorously triaging**, and committing this is the first child sub-plan — everything
in 04 builds on a known-good, committed, green baseline. **Do NOT take "it's just the test" at
face value** (Sub-Plan 02 already hid a real mass-creation bug behind a plausible story); each red

This sub-plan turns the playable-but-raw slice into a balanced, surfaced, replayable game:

### Sub-Plan 01: Integration verification, repair & baseline commit

Secure the uncommitted Sub-Plan 03 work as a known-good, committed baseline before any 04 feature
work. This is the master plan's "assemble the complete loop into a single playable build" step.
- **Verify FIRST, commit SECOND** (do NOT commit a broken tree as a "known-good baseline"): run
  `npx vitest run` (existing 59 Operations/render tests + the new `test/workflows/*` must all
  pass) and `npm run build` (must succeed); start `npm run dev` and confirm a clean boot.
- **Prove the sim is actually LIVE, not just loaded — the `GameLoop` per-frame `try/catch`
  SWALLOWS a throwing `step` and only `console.error`s it, so a broken step is invisible to a
  "no obvious console errors" boot.** The headless smoke (via `window.__chompy`) MUST assert
  *positive change*, not just bounds:
  (a) after N stepped frames a **non-player body's position has changed** (the sim is running);
  (b) setting `input.inputState.up=true` for K frames moves the player in **−y** (controls work)
      and releasing it lets the player coast (no drag);
  (c) a **hand-placed overlapping pair** resolves — a collision counter (`world.counters`)
      increments (collisions fire);
  (d) **spy on `console.error`** across the run and assert **zero** frame errors were logged
      (this catches a silently-throwing step the try/catch would otherwise hide);
  (e) bounds still hold: body count ≤ `MAX_BODIES`, particle/shock pools ≤ caps, all positions
      finite, density ~`POP`;
  (f) **mass-conservation guard:** total field mass is **non-increasing** across N frames within
      dust tolerance (gravity/collisions never CREATE mass). Bounds + finiteness alone won't catch
      a slow mass drift that stays under `MAX_BODIES` — this dedicated assertion does.
- **Triage the 3 known failing `test/workflows/*` tests with prove-code-vs-test discipline**
  (per project rules: a red is EITHER a real bug OR a wrong test — decide which by evidence, never
  by assertion). **COMMON ROOT CAUSE (verify this FIRST):** the three tests seed body
  positions/velocities but do NOT model SimStep's **step-4 integrate pre-pass** — `gravityAccel`
  + `integrate` mutate every body BEFORE the collision loop, so the values entering collisions are
  NOT the seeded ones. Two concrete effects: (a) `integrate` clamps drifter speed to
  `VEL_CAP=4.0`, so a seed like `A.vx=-5.2` actually closes at 4.0; (b) gravity nudges positions by
  sub-pixel/few-px amounts that can flip a pair across an overlap or closing-sign boundary. Trace
  the post-integrate state of each seed before blaming any Operations file.
  - For EACH red, determine whether the *production code* satisfies the *intended invariant* (not
    the literal assertion). If the code is correct and the test's literal/seed/threshold is wrong,
    fix the TEST to assert the invariant and re-confirm it's still falsifiable (break code → red →
    restore). If the code is wrong, fix the CODE.
  - #1 power-law: `P(below MASS_MIN+0.05*MASS_SPAN) = 0.05^(1/POWER_EXP) ≈ 0.49` for
    `POWER_EXP=4.2` — the `>0.5` literal is mis-derived. Assert a **bound derived from
    `POWER_EXP`** (or a median/quantile invariant), NOT a fresh magic literal (so it survives a
    04.04 `POWER_EXP` retune).
  - #2 dead-guard: the no-mass-inflation invariant already passes; the `bodies.length<3` ("a merge
    fired") secondary likely fails because gravity flips the static seed to *separating* at
    sub-pixel scale → SLOW+separating de-overlap instead of merge. Re-seed with a genuine small
    **closing** velocity (so gravity drift can't flip the sign) and **instrument which regime
    branch the first pair actually takes** rather than assuming; do NOT delete the assertion.
  - #3 fracture mass-conservation: **highest-scrutiny, but the likely culprit is the TEST seed, not
    `collision.js`.** A trace of the apply/deferred-buffer path shows it conserves to ~1e-13
    (e.g. 200+200 comparable fracture → 328 surviving + 60 bystander = 388); the red is most
    likely a near-miss (B/C ~1–2px from the overlap threshold after integrate) flipping an extra
    pair in/out. FIRST trace the post-integrate positions; reconcile the expectation against the
    actual pairs that resolve. ONLY if the SimStep APPLY path genuinely drifts surviving mass
    (independent of seed) is it a real SimStep bug to fix — prove it with a seed-independent
    mass-conservation assertion before touching code.
  - **Note for all `test/workflows/*` seeds:** a seeded drifter speed above `VEL_CAP` is a fiction
    (integrate clamps it) — assert/comment that seeded speeds are ≤ `VEL_CAP` to be meaningful.
- **Repair any other broken edge, with a hard bound:** fix ONLY what's needed to reach green
  tests + clean build + the positive-change/zero-error boot. If repair beyond the 3 known reds is
  more than a few small fixes (≈3 attempts or a non-trivial rewrite), **STOP and escalate to the
  user** (per the project's 3rd-attempt rule) rather than open-endedly repairing inside a
  "verification" child. Do NOT add 04 features here.
- **Commit** the verified tree (03 work + any verification repairs). **Separate commits for code
  fixes vs. test recalibrations** so the baseline diff is auditable — never bundle a behavior
  change with a test change in one commit (that conflation is exactly how the Sub-Plan 02 mass bug
  hid). Label clearly. No push. (Last known-green commit is the Sub-Plan 02 hardening commit,
  which predates all of WorldManager/SimStep/Input — the only rollback point, so reaching green
  here matters.)
- **Verification:** green `vitest run`, successful `build`, clean dev boot, the positive-change +
  zero-console-error smoke assertions pass, working tree clean.

  rank, and the growth indicator live every frame; fix any field that reads stale/zero. Counters
  come from `world.counters` (use `?? 0`).
- **Debug overlays — OPTIONAL (the master mandate says "optional"); cut if time-constrained.**
  If kept, a **read-only** Connector (a Pixi `Graphics` layer under `world`, off by default,

  bodies) and (b) collision-regime classification (color nearby pairs by `merge`/`chip`/

  O(n²) pass on top of `SimStep`'s existing one (~140² pairs). Instead **thread `SimStep`'s
  already-computed `acc` array out** (e.g. store it on the world / `window.__chompy` for the
  frame) and have the overlay read it. The overlay must not mutate sim state — verify by
  asserting body state is identical for a stepped frame with the overlay enabled vs disabled.
  If this threading is awkward, **cut the overlay** and reallocate the effort to rigorous 04.01
  verification — it is a dev aid, not a master exit criterion.
- **Non-goals:** no physics edits, no tuning, no restart logic, no game-over copy (that's 04.03).
- **Verification:** unit-test the VM mapping (counters/growth/rank correct for a constructed

### Sub-Plan 03: Lose / restart flow

Make death legible and the game replayable without a page reload.
- **Game-over — this is a real `Hud.js` + VM change, not just copy "polish"** (recon: `Hud.js`
  currently does NOT capture `#otitle` and HARD-CODES the `#osub` string in the Connector,
  reading only `vm.eaten`). Do: (1) capture `#otitle` in the `Hud` constructor; (2) widen the VM
  dead-branch to carry `{title:'Consumed.', summary}` built from `world.counters` (final mass +
  eaten/merged/shatter); (3) have `Hud` render `otitle`/`osub` FROM the VM. Drive overlay
  visibility from `vm.mass==null` ONLY (the already-live path) — no separate one-shot flag.
- **Restart (in-place, replace `location.reload()`)** — a handler owned by `main.js`. The state
  inventory MUST be complete (an incomplete reset is worse than reload):
  - `world.spawn()` (rebuild bodies + player), `world.counters` → zero, `Body._resetIdCounter()`,
    re-center `cam` on the new player, respect the degenerate-canvas guard.
  - **Reset input state** (`input.inputState` all-false, e.g. an `input.clear()`) — otherwise a
    player who dies holding thrust and mouse-clicks "Play again" keeps thrusting (the blur handler
    won't fire on a button click). **This is the highest-risk restart bug.**
  - **Clear FX pools** explicitly — `spawn()` does not touch them; add a `WorldManager.clearFx()`
    (or `world.particles.length = 0; world.shocks.length = 0`) so death debris doesn't leak in.
  - **Do NOT imperatively hide `#over`.** Just make the player exist again and let the next
    `buildHudVM` (player present → `mass!=null`) hide it — driving it both ways is the desync
    vector. Wire restart to the `Play again` button AND a key (e.g. `R`).
- **Invariant to preserve:** the player is NEVER culled (only collision-killed) — confirm
  `cullAndRespawn` still exempts `player` and the opening is survivable. (Note: mid-frame player
  death is already safe — `SimStep` re-resolves the player post-commit each frame and caches no
  stale ref — so no work needed there.)
- **Non-goals:** no tuning.
- **Verification (falsifiable):** unit tests — restart reseeds a fresh world (player at origin,
  counters zeroed, population ~`POP`, ids reset, **`input.inputState` all-false**, FX pools empty)
  and is idempotent; overlay shows iff `vm.mass==null` (keep player → hidden; remove → shown);
  the dead-branch VM `summary` contains the counts (break it → assertion fails). Manual: die while
  holding thrust, click Play again, confirm the fresh player does NOT auto-thrust and no leftover
  bodies/FX/counters.

### Sub-Plan 04: Tuning pass & end-to-end verification

Balance the named constants for the calm, survivable, non-runaway feel, then verify the whole game.
- **Tuning (config-value edits only, `src/config.js`):** balance against the prototype feel —
  `GRAV_EXP`/`GRAV_SOFTEN`/`GRAV_ACC_CAP`/`VEL_CAP` (gravity bends paths but never runs away),
  `V_MERGE`/`V_FRACTURE`/`RESTITUTION`/`SAME_SIZE` (the three regimes each reachable and visibly
  distinct), `FRAG_MASS_LOSS`/`ABSORB_BASE`/`DIMINISH_SCALE` (growth is rewarding but not a
  snowball), `POP`/spawn-ring/`POWER_EXP`/`MASS_MIN/SPAN`/`SEED_SMALL_GUARANTEE` (density +
  small-weighted distribution). Record before/after values and the rationale.
- **Guard against test breakage — respect the FROZEN Operations tests (you may NOT edit them).**
  The `test/operations/*` suite is mostly *symbolic* (it builds inputs from `V_MERGE`/`V_FRACTURE`/
  `SAME_SIZE`/`FRAG_MASS_LOSS`, so they move with a retune) but it bakes in **invariants** the
  retune must not violate. Treat these as **hard constraints on the tuning range** (a retune that
  breaks one deadlocks 04 — you can't fix a frozen test):
  `V_MERGE < V_FRACTURE` **with comfortable margin** (the matrix test puts a "between" midpoint
  `(V_MERGE+V_FRACTURE)/2` in the chip band — keep the gap wide enough that it can't round onto a
  boundary); `SAME_SIZE > 1`; `0 < FRAG_MASS_LOSS < 1`; `GRAV_EXP > 1`; `M_REF > 0`. BEFORE
  editing a constant, grep `test/operations/` for symbolic references and confirm the retune
  respects their implied ordering.
  The "convert a literal-pinned test to an invariant assertion" tactic is **legal ONLY for the
  editable `test/workflows/*` tests** — never for frozen `test/operations/*`. Never weaken a
  falsifiable test to pass a retune. Re-run the FULL suite after each change.
- **End-to-end verification (the master plan's exit criteria):** dev server play (or, since
  chrome-devtools MCP was unavailable in prior sub-plans, manual play + `window.__chompy` headless
  assertions): each collision regime is reachable and visibly distinct (slow merge, medium
  chip+bounce, fast fracture into real bodies); exponential gravity visibly bends drifter paths
  without runaway; stable frame rate over a multi-minute session with no freeze and no
  NaN/explosion (assert via stepped headless run: bounded body/particle counts, all-finite
  positions over thousands of frames); HUD reflects live state; lose/restart works.
- **Final commit** of the tuned, verified build. No push.
- **Verification:** full `vitest run` green, `build` succeeds, the end-to-end checklist above
  passes, before/after tuning table recorded.
